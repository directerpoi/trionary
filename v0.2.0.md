You are a compiler engineer and language designer.

I have built Trionary v0.1.0, a minimal programming language in C with:
- Arithmetic execution
- Variables
- Pipeline system (lst, whn, trn, sum, emt)
- Single-pass execution model
- CLI tool: tri run file.tri

Now I am upgrading to v0.2.0.

Your task is to VALIDATE the upgrade plan, not generate random features.

--------------------------------------------------

UPGRADE GOAL (v0.2.0)

Enhance usability WITHOUT breaking v0.1.0 behavior.

--------------------------------------------------

PROPOSED FEATURES

1. Allow variables inside pipeline transforms:
   Example:
   a = 10
   lst [1,2,3]
   | trn *a
   | sum
   emt

2. Improve error messages:
   - Include line number
   - Include expected token
   - Be human-readable

3. Support multiple pipelines per file:
   Example:
   lst [1,2,3] | sum
   emt

   lst [4,5,6] | sum
   emt

--------------------------------------------------

STRICT CONSTRAINTS

- DO NOT break existing syntax
- DO NOT redesign grammar
- DO NOT introduce VM or bytecode
- DO NOT add complex features (functions, modules yet)
- Maintain single-pass execution
- Maintain minimal keyword set

--------------------------------------------------

VALIDATION TASKS

Analyze and answer:

1. Is this upgrade backward-compatible with v0.1.0?
2. Does it increase complexity significantly?
3. Are there hidden parser or execution risks?
4. Does it affect performance (especially pipeline execution)?
5. Are there better alternatives with less complexity?
6. What exact parts of the system must change? (parser, exec, symbol table)
7. What edge cases will break this upgrade?
8. What test cases must be added?

--------------------------------------------------

REQUIRED TEST CASES

Generate tests for:

- variable in trn (*a, +b, etc.)
- undefined variable inside pipeline
- multiple pipelines in one file
- malformed pipeline syntax
- mixed arithmetic + pipeline

--------------------------------------------------

OUTPUT FORMAT

Provide:

1. Validation summary (safe / risky)
2. Required code changes (file-level)
3. Edge cases list
4. Test cases
5. Final recommendation (GO / MODIFY / REJECT)

--------------------------------------------------

DESIGN PRINCIPLE

Stability > Features  
Clarity > Flexibility  
Execution > Abstraction  

--------------------------------------------------

Now validate the Trionary v0.2.0 upgrade plan.
